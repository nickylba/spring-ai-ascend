# AI 时代的工程运行模式（spring-ai-ascend 实践版）

本文整理 PR #179 交付前后关于"如何用 AI 持续开发并交给 5 人小组维护"的
系列讨论结论。每条原则都标注了本仓库的实证来源——这些不是理论推演，
是过去 48 小时内逐条付过学费的经验。

---

## 1. 治理分层：从"两层"到"三层"

架构师组的初始方案是两层：架构层（人类主导，MD 文档 → 4+1 视图 +
harness/测试计划）+ 自动化层（graphify/swagger 类工具生成 + 人与模型
事后校验修改范围）。

**结论：方向正确，但两层之间存在空隙——必须补入"机器可执行的边界
守护"作为中间层：**

| 层 | 内容 | 性质 | 本仓对应物 |
|---|---|---|---|
| 意图层 | 人类架构文档（粗粒度：平面划分、依赖方向、不变量） | 慢变、人审 | `workspace.dsl`、L0/L1、ADR |
| **法律层** | 结构守护 + 契约测试 + 门禁，每次提交自动执行 | **真正"限制模型"的那一层** | ArchUnit、`gate/`、奇偶校验 |
| 事实层 | 从代码生成的结构/接口文档 | 零维护、不撒谎 | facts 提取器、`architecture-graph.yaml` |

空隙的证明：MD 文档约束不了模型（只是输入），生成式文档只描述现状
（不阻止越界）。没有 enforcer 的文档边界必然腐烂（§9 案例 1）。

**文档即愿望，测试即法律。**

两个落地要点：
- 守护用 **deny-by-default 白名单**，不用黑名单——黑名单会因新包出现
  而过时漏判（§9 案例 4）；
- Swagger 类工具覆盖不了非 HTTP 边界（引擎 SPI、wire 序列化方言）；
  接口签名完全合法、行为却错的 bug 要靠 **wire 级契约测试**拦（§9 案例 2）。

## 2. 并行迭代的"共振"问题与六个阻尼机制

共振的根源不是并行本身，而是**多路变更并发写共享面**（接线文件、POM、
契约文档、生成物）。实证场：本分支 15+ 批次、单批最多 5 路并行、
约 80 个提交全程绿色。起作用的机制：

1. **事前排他域划界**——每路工作开工前声明 OWNERSHIP/FORBIDDEN 文件域，
   把合并冲突变成设计时决策。仅有的碰撞全部发生在未划清的共享文件上；
2. **接缝先行**——B 依赖 A 的新接口时流水线化（A 落地 → API 摘要交接 →
   B 以代码为真相源），不硬并行；
3. **基线 = 不变量集合，不是冻结的提交**——任何通过门禁+测试的状态都是
   合法基线，"稳定"靠自断言而非冻结。这是法律层对敏捷的核心价值；
4. **小批次 + 每批全量绿 + 垂直切片**——坏状态永不过夜；SPI+实现+测试
   同变更落地（零实现 SPI 会腐烂成幽灵契约）；
5. **生成物单写者**——facts/基线/架构图只由集成者在批次收尾统一重生成，
   并行工作者只读；
6. **变更成本分级**——共享面调高（冻结+同提交评审提案），叶子代码调低，
   变更流量自然流向低耦合区。新能力一律加法式+默认关。

效率结论：这套约束不是用效率换稳定——划界消灭合并地狱、自断言基线消灭
排队等待、机器裁决消灭争议挂起。真正的效率杀手是返工和等待。

## 3. 状态而非记忆：注册表的正确形态

"共享记忆缓解共振"的说法需要修正两处：

**其一**，共享记忆只治"认知共振"（重复推导、口径漂移、重开已决议题），
治不了"写共振"（并发写共享面）——后者是控制面问题，解药是 §2 的机制。
而且**腐烂的共享记忆 = 共享幻觉**，比没有更危险（§9 案例 1）。

**其二**（owner 的修正，成立）：协作需要的是**准确的全局状态**，
"记忆"一词把模糊性合法化了。但"注册表 + 要求每次改动同步更新"的执行
形态会重蹈腐烂覆辙——本仓早有这个注册表且烂过。正确机制三条：

1. **能生成的绝不手写**——依赖图/接口签名/拓扑从代码静态提取（带
   commit 戳），注册表的大部分应是代码的投影；
2. **必须手写的（意图：允许依赖、semver 承诺、所有权）每条配奇偶
   校验器**——声明 vs 现实由门禁逐条比对，不一致 = CI 红。
   "必须同步更新"作为纪律必然失效，作为门禁才成立；
3. **按所有权分片**——每模块一个 metadata 文件（与 CODEOWNERS 对齐、
   并行零冲突），全局图由工具汇总。单一全局注册表文件是新的写热点。

补充两条边界：
- 注册表只覆盖**结构状态**；**行为状态**的载体是契约测试（§9 案例 2 在
  任何接口注册表里完全不可见）；**决策状态**的载体是 ADR（不可变史）。
  三者生命周期不同，不可混入一个"知识库"；
- 粒度钉在模块级公开缝，不下探内部——重构 8 个大类注册表零变更，
  恰证粒度正确；登记到类/方法级，注册表自己就变成振源。

> 类比修正：Windows 注册表（手写、孤儿键泛滥）是反例；
> 数据库系统目录 / lockfile（引擎自己读它才能运转，不可能撒谎）是正例。

本项目最好的共享状态是**测试套件本身**——可执行的团队共识，
不可能静默过时（过时即红），对人和模型同等约束。

## 4. 仓库结构：法律与代码必须同仓

仓内三类内容不可混为一谈：① 法律层（构造性耦合于代码）、
② 决策与架构史（语义耦合）、③ 通用引擎（弱耦合，才是"开发平台"）。

**①② 不能拆**：防腐体系依赖**同提交原子性**——冻结面修改 + 评审提案
同提交、metadata/目录与代码 lockstep 由门禁在同一棵树上比对。拆仓 =
版本偏斜矩阵 + 把"必须同步"降级回人类自觉，漂移病跨仓复发。
ArchUnit 守护本就是模块内测试代码——没人会提议把测试拆到另一个仓。

**③ 可拆，但按触发条件**（满足任一再动，当前三条都不满足）：
出现第二个项目想复用 / 治理团队与产品团队分离 / 产品开源而治理内部。
拆法是**引擎/实例分离**：通用框架出去成版本化工具，本仓保留规则实例
并钉版本消费。为一个消费者建平台是反模式。

客户观感问题用**发布面与仓面分离**解决（Maven 只发布模块工件+手册），
比动仓结构便宜两个数量级。

## 5. 组织阵型：四平面 × 主备（契约属主版）

5 人按四概念平面分工，每区双人覆盖（bus factor ≥2）：

| 人 | 主（契约属主 + 具名担责） | 备（指定反方 + 保温） |
|---|---|---|
| A | agent-runtime + agent-bus | runtime.llm.gateway |
| B | runtime.llm.gateway | agent-runtime + agent-bus |
| C | agent-service + starter | 两个 client |
| D | agent-sdk | agent-service + starter |
| E | 两个 client | agent-sdk |

- **主**：本区契约（SPI/wire/属性/不变量）的裁决权；评审对象是
  **证据包**而非逐行 diff；本区守护与 metadata 分片的保鲜责任；
  监管要求的具名负责人（能向审计员做契约级演示）。
  主不是写锁——人类用评审路由，不用排他写入（与 AI agent 的硬隔离
  刻意不同：agent 上下文短命不可对话，人类长命可评审）；
- **备**：高危变更（共享面/安全边界/冻结面）强制双签；**保温义务进
  迭代计划**（每迭代在备区动一次手）——这是退出选项的保费，不是吸收
  手段；主缺位时全权代行；
- **共享面集成人**（root POM/BoM/facts/台账）按迭代轮值，不设固定主；
- 半年主备对调一次，防知识黑洞。

裁决升级路径：区内主拍板 → 跨区两主商定 → 平台级不变量才全组 + ADR。

## 6. AI 开发模式：瓶颈搬家与契约拥有模型

### 6.1 问题的重定义

AI 介入后瓶颈从"写代码"搬家到"人的吸收/验证带宽"。**没被人拥有的
代码不是资产，是库存**。延期的真因是计划只计产码（已经不要钱），
没计吸收（现在的关键路径）。

### 6.2 关键追问：人一定要吸收代码吗？——不一定

逐项检验"读懂代码"服务的功能：修改能力、验证评审、连续性——
**均可交给 AI**（法律层兜底 + 状态纪律保证 AI 可被仓库状态重新实例化）。
不可让渡的只剩三项人类持仓：

1. **责任归属**——监管地板：每个安全控制要有具名人类负责人（银行业
   外部约束，零讨价空间）；
2. **目标与取舍裁决**——目标源自组织，AI 只能提案；
3. **证据裁判**——决定何时不信 AI、强制启动实证。注意：纠错的终极锚
   不是人脑也不是另一个 AI（agent 会继承同一错误框架），而是
   **与现实接触的可执行验证**（§9 案例 3 靠裸线探针而非任何评审打破）。

### 6.3 契约拥有模型 → 风险定价的深度函数（小组质疑后的收敛版）

小组质疑（成立并被采纳）：高危区域的"契约级理解"不够格做归因和担责
——租户隔离出事时，只能复述手册的负责人既判不动 AI 修复方案的成色，
也无法向监管解释根因；且 AI 的最小化完成倾向 × 低人类纵深 = 高危
遗留事项的复利积累。因此"契约拥有"修正为**风险定价的深度函数**
（深度 ∝ 爆炸半径 × 不可逆性）：

| 区域 | 人类理解深度 | 理由 |
|---|---|---|
| 安全边界（租户隔离/JWT/路由授权）、资金路径、不可逆操作 | **L3 实现级**——能徒手改、能独立审 AI 的修复 | 担责必须实心；AI 偷懒后果不可逆 |
| Run 内核、wire 契约 | L3（主备至少一人） | 归因分析的必经之路 |
| 一般业务逻辑、适配器 | L2 契约级 + 质询能力 | 法律层兜底，出事可调查 |
| 胶水、示例、生成物 | L1 行为承诺级 | 同信任未读源码的第三方库 |

高危区的可寻址落地：`docs/governance/risk-register.md`（R1–R5，
精确到文件，与 CODEOWNERS 双签段逐行对应）+ 每区一份深读导引
（`docs/deep-dives/`）。

**AI 偷懒对冲清单**（最小化完成/提前宣告胜利/选便宜解释是真实倾向，
本仓自证：HITL 修复的第一刀选了最便宜的一行改动且失败）：
对抗验证以反驳成功为奖励、守护测试自证非空虚、证据准绳强制实证探针、
独立评审、人持有的停机规则。对冲降低残余风险但不归零——
这正是高危区保持 L3 的理由。

其余结论保持：人吸收的对象主体仍是承诺面（手册、契约目录、不变量
代数、ADR）；团队角色仍是"**立约者 + 裁决者 + 担责者**"；
全员必备的是**调查素养**（会质询、会下钻、会判证据）——"懂"不是
库存而是能力，HITL 缺陷调查中对 SDK 内部的理解就是按需构建的
（台账→线缆探针→字节码的定位级联），预存的理解同样会过期。
三笔保险费不省：监管地板的具名负责人、主备保温、防判断力萎缩。

### 6.4 节拍规则（修正后）

- **契约面变更**：小步走、人真读、批次上限按人眼可读计（≤400 行级）；
- **实现层变更**：AI 放开跑、不限行数、按**证据成色**验收
  （测试覆盖了哪些不变量 / 对抗验证做没做 / 跨边界行为有无实证探针）；
- **WIP 上限按"未立约的契约变更数"记**；到顶该区 AI 停产转讲解员/
  出题人（把 AI 的富余速度调头去攻吸收瓶颈）；
- **完成定义**：特性完成 = 契约更新 + 证据绿 + 担责人签字；
- 证据准绳成文进 PR 模板——"判证据"是团队的新技能，需要工作底稿。

## 7. 交接方案 v2（一页版）

1. 合入 PR #179（小组的"第 0 天"；长期不合的大分支是共振的最大放大器）；
2. **前提项（阻塞）**：法律层全量进 CI——CI 已有门禁/自测/baseline
   守卫，缺口是**场景套件**（示例模块默认 skipTests=true，行为锚此前
   在 CI 不设防）和 **facts 一致性**（重生成后 diff 必须干净，
   `-I` 过滤哈希行）。**信任的全部重量在它身上，不闭环即裸奔**；
3. **立约+深读冲刺**：普通区出口 = 讲清承诺 + 用证据准绳判一份真实
   AI 提交（3-5 天）；高危区（risk-register R1–R5）负责人出口提高到
   "能独立审一份 AI 对该区的修复"，用深读导引压缩路径（加 2-3 天）；
4. CODEOWNERS 按契约属主路由 + PR 模板带证据清单；
5. 常态节拍按 §6.4；止损规则机器执行；
6. 开发体验三小件：`verify.sh`（一条命令 = CI 同款）、快慢分层
   （内环模块级秒回，外环全量分钟级）、新人路径（手册 → quickstart →
   跑场景套件 → 写一个新场景）。

刻意不做：gitflow/develop 分支、更多冻结面、重型流程工具。

## 8. 代码度量的诚实记录（供排期参考）

"sanity 重构"一轮的真实账目：圈复杂度全部降到 ≤10、最长方法 73→30 行、
8 参数链收敛为载体 record——但**总行数 +228（约 +8.6%）**，因为提取
方法/协作者有签名与 javadoc 的固定开销。"可读性"与"总量瘦身"是两类
目标、两类手段（拆分型 vs 消重型），要分开立项。全仓 8 模块生产代码
约 1.5 万行（对照 AgentScope Runtime 约 2.7 万行）。

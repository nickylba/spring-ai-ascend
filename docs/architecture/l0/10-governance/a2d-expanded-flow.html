A2D v2 · 4+1 为枢纽

人类做架构,机器守边界,AI 跑实现

4+1 是持续维护的「架构基线」,不是每轮重建的。每轮需求产出一份「架构更新稿」,人类只审核更新信息(delta);中间 AI 自主实现,围栏防止破坏基线。人类做架构、机器守边界、AI 跑实现。

  • 人类:做架构、做裁决
  • AI:自主实现
  • 机器:守边界(gate + 对账)
  • 闸门:人类必须过的关卡

核心心智:三个角色各司其职

AI 写代码的能力已经很强,瓶颈不在实现,而在大型架构判断。所以把「实现」整个交给 AI,把「架构」牢牢交给人类,把「守边界」交给机器——三者构成一条不互相拖累的链。

● 人类

做架构

维护 4+1 视图,把控方向、模块边界、状态 owner、契约、部署约束和风险。决策的是「这个架构在业务和组织上站不站得住」。

不写代码、不拆切片、不逐行审 diff。
● 机器

守边界

把 4+1 的每条约束转译成实时 gate 和验收对账。AI 破坏即拦,漂移即暴露,替人类守着边界。

不创造架构语义,只投影、拦截、对账。
● AI

跑实现

在围栏内自主派生边界、拆切片、编码、测试、生成证据。碰到 4+1 不合理才升级打断人类。

不自行扩大边界,不偷偷改架构。
关键认知:4+1 是持续维护的「架构基线」,不是每轮重建的。每轮需求真正产出、需要人类审核的,是对基线的「架构更新稿」(delta)—— 人类审的是更新信息,不是整个架构。这让闸门负担随基线成熟而下降:基线越稳,每轮 delta 越小,审核越轻。

总体流程:两道闸门,中间全自动

横向滚动。4+1 是持续维护的架构基线;两个深色加琥珀框的节点是人类必须过的「闸门」,其中闸门 1 审核的是本轮对基线的「架构更新稿」。闸门之间 AI 自主跑,gate 实时守;升级口是 AI 自主区内唯一中断点。

架构更新审核:闸门 1 到底在做什么

4+1 是持续维护的架构基线,不是每轮重建的。这一闸门审核的不是整个架构,而是「本轮要对基线做的更新」—— 所以人类审的是更新信息(delta),负担随基线成熟而下降。需求到更新稿这段过程 AI 自动完成,但它是必要环节,不能省。

读取现有 4+1 基线 + 代码现状扫描
→
结合本轮需求,分析冲击哪些架构事实
→
AI 产出架构更新稿(新增 / 修改 / 废弃)
→
人类审核更新信息
→
通过则作为本轮目标(待合并,不写基线)
为什么更新稿不立即写基线:实现过程可能推翻设计思路(假设不成立、需要调整)。更新稿是「意图」,只有实现验证、根据实际落地后,才在末端固化进基线。实现期间 gate 和 graphify 按「已固化基线 + 本轮更新稿」对账,但不污染已固化基线。

AI 产出物要求 ——「架构更新稿」必须包含

字段 要求
delta新增 / 修改 / 废弃了哪些架构事实(模块边界、状态 owner、契约、依赖方向、部署约束…)
理由为什么需要这个更新,关联到哪条需求或哪处代码现状
冲击的不变量动了哪些既定不可变项,是否可接受
gate 变更哪些 gate 的期望值要跟着改,新的期望是什么
升级标记有没有需要人类额外裁决的硬约束(越界 / 改 owner / 破坏契约)

围栏机制:4+1 不是给人看的文档,是可执行的约束

写在文档里的「模块边界」拦不住 AI。要真正限制破坏性修改,4+1 必须能落地成自动检查。三个环节其实是同一组架构约束的三个投影——蓝图是期望,gate 是实时守,对账是事后验。

① 蓝图

4+1 视图

人类维护的「期望」。写明模块边界、状态 owner、依赖方向、契约、部署约束。这是人类把控架构的唯一界面。

② 实时守

gate 执行

把蓝图转译成实时拦截。AI 写代码时 ArchUnit、依赖 gate、schema check 即时拦下破坏,不需要人类看 diff。

③ 对账

graphify / OpenAPI

验收时从代码反推现实结构,和蓝图做 diff。漂移 = 破坏点,自动暴露给闸门 2 的人类裁决。

4+1 的每一类视图,都对应一类 gate。实现期间 gate 和 graphify 按「已固化基线 + 本轮待合并更新稿」对账(本轮目标);只有末端固化后,基线才真正吸收本轮已验证事实。graphify + 4+1 配对,就是「架构围栏」的自动对账机制。
4+1 视图 围栏守什么 对应的 gate
逻辑视图模块职责边界ArchUnit / rg gate:某包不许调用某包
开发视图分层依赖方向依赖方向 gate:禁止反向依赖
进程视图状态 owner、并发模型状态归属 contract test / drift check
物理视图部署单元、配置部署 / 配置 schema check
场景视图关键路径行为wire probe / 端到端测试

4+1 的双面:人类的界面,机器的蓝图

同一份 4+1,对人类和对机器承担两种角色。它既是人类把控架构的全景基线,也是机器守边界的约束源头——而且它是持续维护、增量演进的,不是每轮重画。

给人类:持续的全景基线

每轮审核的是基线的更新(delta),不是从零把控整个架构。决策的是更新在业务和组织上站不站得住。

  • 一眼看清模块边界、状态归属、依赖方向
  • 识别本轮更新会冲击哪些不变量
  • 决定升级口和更新稿里的问题怎么裁决

给机器:持续生效的约束蓝图

实现期间围栏对齐「已固化基线 + 本轮更新稿」(目标);末端固化后,基线才吸收本轮已验证事实。

  • 代码边界说明 =(已固化基线 + 更新稿)∩ 代码扫描 → 自动派生 writable
  • 每条约束配置一个 gate 的期望值
  • graphify 反推代码现实,与「基线 + 更新稿」对账

升级口:围栏不是死墙

4+1 是静态快照,架构是演进的。如果 AI 每次都必须严格符合 4+1,合理改进也被挡死,架构就僵化了。所以围栏要留一个口——这是 AI 自主推进过程中,唯一需要打断人类的地方。

什么时候触发

越界
实现需要改动当前 4+1 基线之外的模块或路径
改 owner
需要改变状态 owner 或跨模块控制权
假设崩
代码证明 4+1 基线的某条假设不成立

AI 怎么做

不硬闯
gate 会拦,闯不过去
不偷改
偷改会被 graphify 对账暴露,破坏基线
而是
升级回闸门 1,调整本轮待合并更新稿,由人类重新审核

0→1 与 1→3:围栏的成熟度不同

这套模型的核心收益在增量开发——已有 4+1 基准、gate 已在守、graphify 能对账,AI 自主区可以开到最大。从零新建时人类在闸门 1 的负担更重。

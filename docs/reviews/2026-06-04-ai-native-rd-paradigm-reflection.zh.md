# AI 原生时代的研发范式 —— 一次组织级回顾与运营模型

Date: 2026-06-04
Status: reflection / 待 owner 评审
Owner（方向 + 批准）: repository owner
触发: 一个月"架构设计无法闭环" vs 3–5 天"工程闭环"的强烈对照
范围: 本文只覆盖 **A**（范式诊断 + 运营模型）。存量治理/知识资产的处置（**B**）另文。

---

## 0. 这份文档要回答的三个问题

1. 在 AI 原生时代，一个**几十人**的组织，**哪种研发模式**更适合协作？
2. 研发范式到底**变了什么**？变化的机制是什么，而不只是"要敏捷一点"。
3. 最烧心的一个：**大家以怎样的方式"进入"**一个 AI 原生工程？

下面先把事实摆出来，再从机制上诊断，再给运营模型，最后诚实地报这套模型自己的雷。

---

## 1. 事实：两条线

不替任何一方美化，下面每一句都能在 `git log` 里对上。

**线 A —— 一个月的架构设计（5 月）。**
rc1 → rc38 一路的 reviewer 响应、ADR 治理、工程治理、知识治理。终局那个 rebalancing commit 写得很直白：`157 -> 36 rules`；ADR 分四批删了约 **90** 个。也就是说，一个月造出来的治理体系，**最后被它自己删掉了四分之三以上**。这不是意外，是这套打法的**必然产物**（见 §3）。

**线 B —— 3–5 天的工程闭环（6 月初）。**
`feat(engine)` 几乎一天之内把 engine 从模型、事件、SPI、adapter、auto-config 一路建到 real-LLM smoke IT；接着 `align 5 modules to agentscope-runtime and close the end-to-end loop`；再 A2A e2e、多轮 session e2e。关键动作只有两个：
- **对齐到一个外部已验证的 runtime**（agentscope-runtime），而不是从纯逻辑发明一个；
- **先把端到端环路在 samples 里用真 LLM 跑通**，再 outside-in 往外补能力。

**线 C —— 现在这条分支（`agent-runtime-record-reconciliation` / ADR-0159）。**
是拿**架构记录去对齐工程已经建成的东西**。注意权威方向：`reality → record`，不是 `record → reality`。

这三条线拼起来，答案其实已经写在墙上了。

## 2. 一句话的判断

> **旧组织把稀缺资源花在"生产产物"（架构、流程、规约）上；AI 原生组织要把稀缺资源全部搬到"证伪器"上。架构因此从"一份人来捍卫的文档"，变成"CI 每次提交都求值的一个布尔量"。**

本文剩下的一切，都是这一句的推论。

---

## 3. 诊断：架构为什么无法闭环（是机制，不是人）

你说的三个症状——架构师无法达成一致、不停加码导致能力漂移、设计与工程差距巨大——不是三个独立的毛病，是**同一个结构缺陷**的三个投影：**架构文档这个对象，没有成本函数。**

**机制一：无成本函数 → 不可证伪 → 无限膨胀。**
文字是免费的。在一份设计文档里，没有任何东西会推回来说"这条跑不通"。一个没有证伪器的系统，其规模只会单调上涨。你们后来用 gate / 规约 / 门禁试图**人造**一个成本函数——但流程只能约束**形式**（你有没有走仪式），约束不了**有效性**（它到底跑不跑得通）。于是它只增加了摩擦，没增加 grounding，最终 157 条规则里那 121 条自指的仪式被自己删掉。

**机制二：梯度永远指向"更多"——加码是结构必然，不是纪律问题。**
这一点值得单独点破：每一个看设计文档的 reviewer，都被**结构性地激励**去加一条关切——加，显得尽责，且免费，且不会被推回。一份设计文档是一个**梯度永远指向 MORE** 的结构，没有任何力把它往 LESS 拉。所以"不停加码"不是某个架构师不克制，是**只要对象是无 grounding 的文档，任何认真的人都会加码**。你不可能靠意志力或更多治理去对抗这个梯度——那是注定要输的消耗战。**唯一的解法是换掉梯度本身**：让每一个新增都必须"交租"（保持脊柱绿、挂到某个接缝上），梯度就反转指向"最小"。

**机制三：无裁判 → 架构分歧不可判定 → 靠权威或耗尽收场。**
两个架构师在一份无运行裁判的设计上争论，可以**永远**争下去，因为没有 ground truth 可以诉诸。这种分歧最终只能由**权威**（有人拍板）或**耗尽**（吵累了）来终结——两者都留下怨气和漂移。这正是"多个架构师无法达成一致"的根。

而 **AI 原生时代有一个全新的、几乎免费的裁判**：把两种设计各做成一个 spike，看哪个能让脊柱保持绿、哪个更容易挂上接缝。**分歧的解决从"在文档里辩论数周"变成"赛跑廉价 spike 数小时"。** 因为 spike 现在免费了。这是组织级的一条新规则：**用赛跑 spike 取代评审辩论。**

**机制四（设计—工程鸿沟的真名）：它其实是"无 grounding 与有 grounding 之间的鸿沟"。**
逻辑设计运行在一个**无阻力**的空间里（任何东西都可表达）；工程实现运行在一个**充满阻力**的空间里（LLM 就是那样表现、框架就是那个形状、agentscope-runtime 强加了一套分层）。**设计越大、越无 grounding，鸿沟就越宽。** 这就是为什么"从基于逻辑的架构转成现有技术很难"——不是技术不行，是那份逻辑从未与阻力接触过。

---

## 4. 核心反转，以及一个必须正视的事实：AI 是放大器

旧范式全部的合理性建立在一个前提上：**生产很贵**。代码贵、改贵，所以"先想清楚再动手"是对的——measure twice, cut once，因为 cut 贵。重型架构、共识、文档逻辑链，都是在为"昂贵的实现"做前置优化。

AI 原生时代这个前提没了。**生产几乎免费，扔掉也几乎免费。** 当生产免费，瓶颈就唯一地转移到 **grounding / 验证 / 选择**——"在无限便宜的产物里判断哪个是对的"那个成本函数上。

**这里有一个让人不舒服、但必须写下来的事实：AI 不会自动把你推向正确的范式，它只会放大你指给它的那个范式。**
回看线 A：rc1–rc38 那一长串治理仪式，很大一部分正是 **AI 在高速生产 ceremony**。组织一开始把 AI 指向了"生产设计/治理产物"这一侧——于是 AI 把那个**注定漂移的失败模式加速了**。直到线 B，AI 才被指向"生产能跑的代码"这一侧。

含义很硬：**AI 把"选错范式"的代价放大了，也把"选对范式"的收益放大了。所以在 AI 时代，"先把范式选对"不是更次要、而是更高杠杆的决策。** 同一个 AI、同样的算力，指向证伪器就闭环，指向文档就漂移。

## 5. 运营模型：N 人 / M 接缝 / 1 脊柱

把上面压成一个可执行的拓扑。**N 个人、M 个接缝、1 根脊柱。协调成本 ≈ O(M)，不是 O(N²)。**

**柱① 脊柱 = 唯一的真相单位。**
组织里有且只有一个"端到端能跑、且始终绿"的东西（你们现在就是 A2A LLM e2e sample）。它不是"最重要的功能"，而是**那条最薄、却穿过每一个架构接缝的路径**——本质上是"对架构本身的一个持续运行的集成测试"。**架构的有效性 = 脊柱是否绿，由 CI 每次 commit 求值。** 这就是线 A 缺的东西：没有脊柱，架构有效性不可证伪，所以它膨胀。

**柱② 接缝（seam）= 唯一昂贵的动作。**
"先设计"没有死，它被**收缩到接缝上**。规则是：**模块内部你想怎么干都行；接缝是唯一需要跨团队达成一致的东西。** 故意慢、故意贵的，是"改一个接缝"，不是"写模块里的代码"。
而且接缝**不该从逻辑推**，应该**让脊柱跑出来、暴露最小接缝、再冻结**（你们 history 里的 `taskflow SPI freeze`、`EngineDispatchSpi`、`bus.spi.engine` 正是在干这件事）。那一小撮幸存的治理，应当全部集中在**守接缝**，模块内部放开。

**柱③ 切片 = 工作与入场的单位。**
工作不按"层"切（先把 L1 做完整再做 L2——这正是"完整性"逼出 scope 爆炸的地方），而按"能力切片"切：挂到已有接缝、保持脊柱绿、自带一条薄验证。Backlog 是"脊柱还不具备、且被真实 persona 旅程拉动的能力"。**outside-in = 被 `PRODUCT.md` 拉，不被 `ARCHITECTURE.md` 推。** "完整性"这个本能，被"对真实需求的覆盖"替换掉。

**柱④ 人 / AI 分工 = 这套模型里最"AI 原生"的部分。**
走骨架、薄接缝，agile 早有。AI 真正改的是这一条：**AI 在上下文里能装下的"正在运行的系统"，比任何人脑子里能装的都多。** 于是过去人要做的"建立全局心智模型"，现在由 AI **按需**完成（codegraph、对着活代码问"X 怎么工作"）。分工因此重排：
- **人掌管**：意图 / 产品拉力（脊柱下一步该长什么）、接缝决策（少数昂贵的跨团队一致）、以及**对验证的判断**（这条绿，是真绿还是空心绿）。是"做什么"和"这到底对不对"。
- **AI 掌管**：生产（写便宜可抛的内部实现）、局部导航（替人回答"这怎么工作"，于是人不必预先背）、验证的**执行**（跑环路、生成测试、抓漂移）。是"怎么做"和"跑给我看"。

> 最关键的一句：**当生产免费、且由 AI 产出，人不可替代的价值就坍缩到"判断免费产物到底对不对"上——验证判断成了最稀缺的人力资源。** 组织的资历和品味，应该堆在成本函数上（绿意味着什么、脊柱有没有真在压住真实风险），而不是堆在生产上。架构师的角色，从**作者**变成**接缝的园丁 + 脊柱的策展人**。

**柱⑤ 记录退化为投影，而不是规约。**
架构记录（ARCHITECTURE.md / ADR）应该是**从运行系统 emit 出来的滞后索引**，不是人手写的前置规约。你们的 codegraph、architecture-graph-as-index、以及现在这条 `reality → record` reconciliation，都是在用手做一件本该自动的事。目标态：记录是**派生**的，于是它**定义上不可能漂移**。ADR-0159 这种"事后对齐"的痛，在目标态里应当消失。

## 6. 大家以怎样的方式"进入"

这是你最烧心的问题，模型给的答案很干脆：

> **入场物是那根能跑的脊柱 + sample，不是文档语料库。**

新人不靠"读懂全局"进入。他 clone → 本地把 sample 跑绿 → 挑一个接缝内的切片 → 往外扩 → 保持脊柱绿。他的"理解"是**局部的、被验证的**，不是**全局的、靠背的**。

由此推出两条对几十人规模至关重要的性质：
- **入场成本几乎与系统规模无关**（因为入场是局部的）。旧模型里系统越复杂入场越贵——这正是几十人 scale 不上去的根因。
- **入场技能变了**：从"能不能读懂架构"，变成"**能不能把脊柱跑绿、并判断它是不是有意义地绿**"。招聘、上手、晋升，都该照这个新技能重新校准。

## 7. 为什么两次治理探索结构性地失败

**AI 工程治理（严过程件 / 规约 / 门禁）。**
它在自动化旧范式那一招——"用流程当成本函数"。而这一招在生产免费时**本就瞄错了目标**（§3 机制一）。你不是把治理做差了，你是用更大的力气去管控一个**注定漂移的对象**（一个无 grounding 的设计空间）。正确做法：**能表达成运行检查的，就做成运行检查，不要做成需要人记的 rule-card**；治理收缩到"守接缝 + 守脊柱绿 + 极少数无法用测试表达的安全/正确性不变量"。`157 → 36` 是组织在经验上发现这件事；终点很可能比 36 还小。

**AI 知识治理（让很多人基于同一套逻辑体系 / 链路去理解）。**
它的失败可以叫**"地图—疆域坍缩"**。知识治理试图建一张所有人据以导航的共享**地图**（逻辑链、可追溯链）。但在快速变动的系统里，**疆域（运行代码）变得比地图能被维护的速度更快**；而且 AI 现在能**直接读疆域**。当你能直接查疆域时，维护一张独立的地图就是**纯负债**——它只能要么过期、要么冗余。
正确做法：**把疆域做得可读、可查（投资运行系统对 AI 的可导航性），让地图被自动派生（投影）。** "知识"不再是一份人来维护的语料库，而是**对活系统的一种查询能力**。你们 `knowledge/`（非治理、可搜索）的拆分方向是对的，但仍把它框成"语料库"，而不是"对现实的查询能力"——这一步还差半步。

## 8. 这套模型自己的失败模式（诚实报雷）

一个好模型得敢报自己的雷，否则就是又一份无 grounding 的文档。

1. **空心绿**：脊柱过了，但没压住真实风险 → 假信心。这恰恰是为什么"人的验证判断"是最稀缺技能（柱④）——它是这套模型的阿喀琉斯之踵，也正好解释了人的角色为什么是这个。缓解：脊柱必须刻意覆盖真实 persona 旅程里的真实风险，而不是最容易绿的路径；"绿是否有意义"是 senior 的核心职责。
2. **接缝抖动**：如果接缝没真冻住，outside-in 会退化成无休止的 churn。缓解：接缝是那唯一"贵而慢"的东西，改它要走重流程；其余全放开。
3. **无人持有全局 → 涌现性失序**：局部理解人人都有，全局没人扛。缓解：AI 持有全局（codegraph）+ 脊柱即一致性检查。**残余风险**：AI 的"全局"可能是浅的——这一条我不替它遮，需要靠"定期让 AI 做全局体检 + 人对接缝集做策展"来兜底。

## 9. 这一个月并非白费（智力上）

必须把这句写进来，否则就成了"你浪费了一个月"，那既不公平也不准确。

那一个月产出了**领域词汇**（session / task / runtime / engine / suspend-resume / A2A）、**领域理解**、**"什么不该做"的知识**，以及**发现 agentscope-runtime 是对的参照**——正是这些，让线 B 能在 3 天里挑对最小集、迅速对齐。

它白费的不是智力，是**把探索的产出当成了交付的契约**。**那一个月本质上是一个巨大的 spike——而 spike 本该被扔掉。** 错误在于组织试图**保留**它（凝固成 157 条规则、143 个 ADR），而不是榨取其认知、丢弃其形体。

> 给未来的一条纪律：**为探索留预算，但预先承诺把它扔掉。** Spike, don't spec；走骨架先行；文档在运行系统的下游。

---

## 10. 为什么这是"AI 原生"，而不是"敏捷换皮"

Lean / XP / walking-skeleton 早就说过"做最小的端到端"。AI 改的是**约束**，从而改的是**组织拓扑**，不只是迭代粒度：

- **Spike 几乎免费** → 可以**赛跑 N 个 spike** 来判定分歧（§3 机制三）。
- **生产免费** → 瓶颈整体移到验证 → **验证判断从边角活变成核心人力技能**。
- **AI 能持有全局模型** → 人的入场从"全局理解"转为"局部运行 + 验证" → 组织以不同方式 scale。
- **AI 放大你指给它的任一范式** → **"选对范式"成了最高杠杆的决策**（§4）。

敏捷是在"生产仍偏贵、人仍持有模型"的约束下缩小批量。AI 把这两个约束都拿掉了——于是变的是**组织结构**，不只是批量大小。

---

## 11. 通往 B 的桥（占位，另文展开）

如果上面的范式成立，那么存量治理/知识资产的处置（B）就有了判据，而不是凭感觉砍：

- **薄留**：能守接缝、能守脊柱绿、或表达无法用测试表达的安全/正确性不变量的——留，且尽量做成运行检查。
- **改造**：知识体系从"人维护的语料库"改造成"对活系统的查询能力 + 自动派生的投影"。
- **退役**：一切"用流程当成本函数"的过程件、一切要求人预先内化全局的强制阅读、一切无 grounding 的规约——退役。

每一项的判据只有一句：**它是在给运行系统装证伪器，还是在生产又一份会漂移的产物？**

— 完（A 部分）。B 待批准后另文。
